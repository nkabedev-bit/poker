# План работ: Устранение потери данных при отмене вылета (Rollback Fix)

## Проблема
При вылете игрока в таблицу `bounty_log` записывается слепок списка игроков `players_before`. При отмене этого вылета эндпоинт `POST /api/tma/eliminations/[id]/cancel` восстанавливает всю таблицу игроков из этого сохраненного слепка. Если между моментом вылета и моментом его отмены в турнир зарегистрировался новый участник (или админ изменил стек другого игрока), эти новые изменения полностью удаляются и перезаписываются устаревшим снимком.

---

## Предлагаемое решение
Перевести логику отмены вылета на точечный (дельта) откат. Вместо восстановления списка игроков из старого слепка, алгоритм должен модифицировать текущее состояние списка игроков, меняя параметры только тех участников, которых коснулся этот вылет (выбывшего игрока и его киллеров).

---

## Пошаговый план реализации

### Шаг 1. Анализ текущего кода отката
Сейчас в `app/api/tma/eliminations/[id]/cancel/route.ts` используется следующая логика:
```typescript
const updatedPlayers = isTournamentPlayers(typedLog.players_before)
  ? typedLog.players_before
  : getFallbackRollbackPlayers(typedLog, extras.players);
```
Так как `players_before` почти всегда присутствует в логе и проходит валидацию, метод `getFallbackRollbackPlayers` (который и делает точечный откат) никогда не вызывается.

### Шаг 2. Переход на точечный откат по умолчанию
1. В файле `app/api/tma/eliminations/[id]/cancel/route.ts` изменить логику выбора данных для отката:
   ```typescript
   // Игнорируем players_before и всегда применяем точечный откат к текущему списку игроков
   const updatedPlayers = getFallbackRollbackPlayers(typedLog, extras.players);
   ```
2. Аналогичную логику точечного отката применить в `PATCH /api/tma/players/[id]` при обработке экшена `restore_player`.

### Шаг 3. Доработка `getTargetedEliminationRollbackPlayers` в `lib/tma/elimination-rollback.ts`
Убедиться, что функция правильно восстанавливает:
1. Статус выбывшего игрока: `status: "active"`.
2. Поле места: `finishPlace: null`.
3. Число ребаев (если вылет был с ре-энтри, уменьшаем количество ребаев на 1).
4. Возвращает списанные фишки баунти игрокам, которые исключили его (уменьшает их `bountyCount`, `bountyChipsTotal` и текущий `stack` на долю баунти за этот вылет).
5. Mystery Bounty Points (если применимо).
При этом все остальные игроки (включая новых участников, зарегистрировавшихся после вылета) остаются без изменений.

### Шаг 4. Тестирование сценария гонки при откате
1. Зарегистрировать 5 игроков в панели.
2. Провести вылет Игрока №5 (Игрок №4 получает баунти).
3. Добавить нового Игрока №6 в панель или зарегистрировать через бота.
4. Нажать кнопку «Отмена вылета» для Игрока №5.
5. Убедиться, что:
   * Игрок №5 снова стал активен.
   * У Игрока №4 списались баунти и фишки.
   * Игрок №6 **остался в списке участников** и не исчез.
